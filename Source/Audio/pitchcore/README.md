# PitchCore — オフライン ピッチシフトエンジン

尺 (再生時間) を維持したままピッチだけを半音単位でシフトする、オフライン処理専用の
ピッチシフトエンジン。歌い手向け DAW「Utawave」のキー変更機能のために開発した。

- **フレームワーク非依存**: C++17 標準ライブラリのみ (FFT も自前実装)。
  依存はリサンプラの [r8brain-free-src](https://github.com/avaneev/r8brain-free-src) (MIT, ヘッダオンリー) だけ
- **ポリフォニック対応**: カラオケ音源のようなフルミックス素材を想定してチューニング
- **オフライン特化**: リアルタイム制約がない分、品質優先のパラメータ (大きい FFT・高オーバーラップ・2 段処理) を使う
- 実用範囲は **±6 半音** (それ以上も動くが劣化が目立ち始める)

---

## 1. 構成と切り出し方

```
pitchcore/
├── PitchShiftEngine.h     # 公開 API (planar float 入出力)
├── PitchShiftEngine.cpp   # 実装 (タイムストレッチ + リサンプル)
├── RealFFT.h              # 自前の実数 FFT (radix-2, ヘッダオンリー)
└── README.md              # このドキュメント
```

単体ライブラリとして配布する場合は、このディレクトリと r8brain-free-src を持ち出して
`PitchShiftEngine.cpp` を 1 つコンパイルするだけでよい (include パスに r8brain の
`CDSPResampler.h` が見えること)。ホストアプリ側のアダプタ例は `../PitchEngine.h/cpp`
(JUCE の `AudioBuffer` / `File` との橋渡し・約 100 行)。

```cpp
#include "PitchShiftEngine.h"

std::vector<std::vector<float>> out;
const float* chans[2] = { left, right };
bool ok = pitchcore::shiftPitch(chans, 2, numSamples, 48000.0, +3.0, out,
                                [](double p) { /* 進捗 0..1 */ return true; });
```

- 入力: planar float (チャンネルごとの連続配列)。出力は入力と**同チャンネル数・同サンプル数**
- `semitones = 0` は完全コピー (ビット同一)
- `onProgress` が `false` を返すと中断し、戻り値も `false`
- スレッドセーフ (状態はすべてローカル)。1 回の呼び出しは 1 スレッドで実行される

## 2. アルゴリズム全体像

ピッチシフトを「**タイムストレッチ → 再生速度変更**」の合成で行う。`r = 2^(semitones/12)` として:

```
入力 (尺 L, ピッチ f)
  │ ① フェーズボコーダでタイムストレッチ ×r     (尺 rL, ピッチ f のまま)
  ▼
  │ ② リサンプル sr → sr/r、出力を sr として解釈 (尺 L に戻る, ピッチ rf)
  ▼
出力 (尺 L, ピッチ rf)
```

この分解の利点:

- ストレッチ段は**元素材をそのまま解析**できる (先にリサンプルすると劣化した素材を解析することになる)
- ピッチを上げる場合 (r > 1)、リサンプル段が低い方のナイキスト (sr/2r) で帯域制限するため
  **エイリアシングが構造的に発生しない**
- 必要な独自実装がタイムストレッチに集約され、リサンプルは実績ある r8brain に任せられる

### ② リサンプル段

チャンネルごとに `r8b::CDSPResampler24` (srcRate=sr, dstRate=sr/r) を用意し、4096 サンプルの
ブロックで通す。入力が尽きたらゼロを送ってリサンプラ内部の遅延分を吐き出させ、目標サンプル数
(= 入力と同尺) に達したら打ち切る。`CDSPResampler` は開始遅延を内部で除去するため、
出力先頭はストレッチ出力の先頭に対応する (タイミングがずれない)。

## 3. フェーズボコーダ (タイムストレッチ) の詳細

### 3.1 STFT パラメータ

| 項目 | 値 | 理由 |
|---|---|---|
| FFT サイズ N | 8192 (sr ≥ 88.2kHz は 16384) | §5.3 参照。4096 では近接音が劣化する |
| 窓 | Hann (periodic) | 解析・合成の両方に使用 (計 Hann²) |
| 合成ホップ Hs | N/4 (75% オーバーラップ) | ピーク位相ロックとの組み合わせの定番 |
| 解析ホップ Ha | Hs / r (double で保持) | 合成側を固定し解析側を可変に |

フレーム m の解析位置は `ta = round(m·Ha)` (Ha は double 累積)、合成位置は `ts = m·Hs`。
位相伝播には**実際の整数差分 `dt = ta(m) − ta(m−1)`** を使うため、レシオの量子化誤差が
時間方向に蓄積しない (ドリフトゼロ)。

### 3.2 位相伝播 (ピークの瞬時周波数推定)

スペクトルピーク bin p について、bin 中心の角周波数 `Ωp = 2πp/N` からの偏差を
前フレームとの位相差分から推定する:

```
δ  = princarg(φ[p] − φprev[p] − Ωp·dt)     // princarg: (−π, π] へ折り返し
ω  = Ωp + δ/dt                              // 真の瞬時角周波数 (rad/sample)
ψ' = ψ[p] + ω·Hs                            // 合成側の新しい位相
```

定常正弦波ならこの推定は厳密で、出力は入力と同一周波数の連続正弦波になる
(ユニットテストで ±3Hz 以内を検証、実測はサブ Hz)。

### 3.3 ピーク位相ロック (縦コヒーレンス)

フェーズボコーダ特有の「フェージーさ (位相が散ってぼやける音)」は、bin ごとに独立に
位相を進めると **1 つの音を構成する隣接 bin 群の位相関係が壊れる**ことが原因。対策として
領域単位の複素回転 (identity phase locking) を行う:

1. **ピーク検出**: `mag[k]` が ±1 および ±2 近傍より大きい局所最大 (閾値 1e-9)
2. **領域分割**: 隣接ピーク間の振幅最小 bin (谷) を境界とし、各 bin を最寄りのピークに帰属させる
3. **領域回転**: ピークの位相更新量 `θ = ψ' − φ[p]` を**領域内の全 bin に同一適用**する
   (複素数のまま `e^{iθ}` を乗算)。これで領域内の解析時の位相関係がそのまま保たれる

DC (k=0) とナイキスト (k=N/2) は実信号の対称性を保つため回転しない。
ピークが 1 つも無いフレーム (無音) は素通し。

### 3.4 トランジェント検出と「アタック領域限定」位相リセット

位相伝播はアタック (ドラム等) を「過去の位相の続き」として再構成するため、そのままでは
アタックが鈍る。一方で、検出したフレームの**全 bin** をリセットすると、ドラムが鳴るたびに
同時に鳴っている持続音 (コード等) の位相が飛ぶ — 開発中の計測 (§5.2) でこの劣化を確認した。

そこで 2 段構えにしている:

- **フレーム判定 (全チャンネル共有)**: スペクトラルフラックス
  `flux = Σ_k max(0, mag[k] − magPrev[k])` (チャンネル間の最大値) が、直近約 1 秒の
  中央値の 2 倍 かつ 直近最大の 0.1 倍 を超えたらトランジェントフレーム。
  直前のリセットから 4 フレームの不応期を置く。判定をチャンネルで共有するのは、
  左右で別々のタイミングにリセットが走るとステレオ定位がにじむため
- **領域判定**: トランジェントフレームの中でも、位相リセット (`ψ = φ`、解析位相をそのまま使用)
  を行うのは**ピーク振幅が前フレーム比 2 倍を超えて立ち上がっている領域だけ**。
  持続音の領域は通常の位相伝播を続ける

これにより「ドラムは鮮明・持続音は位相連続」を両立する。

### 3.5 時間写像と端のパディング

解析フレーム `[ta, ta+N)` の中心 `ta+N/2` が合成フレーム中心 `ts+N/2` に対応するため、
入力位置 x の内容は出力では

```
y(x) = r·x − (r−1)·N/2
```

に現れる (単純な `y = r·x` ではない)。実装では:

1. 入力の前後に N サンプルの無音パディングを付けてからストレッチする
   (曲頭・曲末も OLA が定常状態で正規化され、端にフェードがかからない)
2. 出力側で先頭 `round(pad·r − (r−1)·N/2)` サンプルを捨てる
   (`pad·r` だけ捨てると (r−1)·N/2 — +3 半音で約 8ms — タイミングがずれる。§5.1)
3. 末尾側のパディングは捨てずにリサンプラへ渡す (ルックアヘッドが実データの無音を読める)

### 3.6 OLA (オーバーラップ加算) と正規化

合成フレームは IFFT 後にもう一度 Hann 窓を掛けて加算する (解析と合わせて Hann²)。
同時に窓の二乗和 `norm[i] += w[i]²` を蓄積し、最後に `out[i] /= norm[i]` で割り戻す
(norm > 1e-4 の範囲のみ)。75% オーバーラップの Hann² の定常和は 1.5 で、
パディングのおかげで実データ範囲は常に定常領域に入る。

### 3.7 自前 FFT (RealFFT.h)

外部依存を無くすため、実数 FFT を自前実装している (`RealFFT.h`、ヘッダオンリー):

- radix-2 Cooley-Tukey (反復・ビットリバース・twiddle 事前計算)
- 実信号 N 点を「偶数番目 + i·奇数番目」で N/2 の複素 FFT 1 回にパッキングし、
  スペクトルの偶奇分解で bins 0..N/2 を復元する標準手法 (約 2 倍高速)
- `forward → inverse` は恒等 (inverse に 1/N スケーリング込み)。
  ユニットテストでラウンドトリップ誤差と正弦波の bin 位置・振幅 (A·N/2) を検証

## 4. チューニング定数

`PitchShiftEngine.cpp` 冒頭に集約してある:

| 定数 | 値 | 意味 |
|---|---|---|
| `kOverlapDiv` | 4 | 合成ホップ = N/4 (75% オーバーラップ) |
| `kFluxRatio` | 2.0 | トランジェント判定: フラックス > 中央値 × この値 |
| `kFluxFloor` | 0.1 | 同: フラックス > 直近最大 × この値 (微小ゆらぎの誤検出防止) |
| `kFluxHistLen` | 47 | フラックス履歴長 (中央値・最大の母集団) |
| `kMinResetGapFrames` | 4 | リセットの不応期 (フレーム数) |
| `kAttackRise` | 2.0 | 領域リセット条件: ピーク振幅が前フレーム比この倍率超 |

FFT サイズは `shiftPitch()` 内 (`sampleRate >= 88200 ? 16384 : 8192`)。

## 5. 設計判断の経緯 (開発中に計測で見つけた問題)

合成検証曲 (和音 + ベース + メロディ + キック/ハット) を作り、従来利用していた
外部エンジンと客観比較しながら以下を修正した。**いずれも聴感に出るレベルの問題で、
かつユニットテスト (純音) では検出できなかった**。同種の変更をする際は §6 の検証を必ず通すこと。

### 5.1 タイミングが (r−1)·N/2 ずれる

オンセット検出で出力が +3 半音時に約 8ms 早いことを発見。原因は時間写像を `y = r·x` と
仮定してパディングを `pad·r` だけ捨てていたこと。フレーム中心対応の正しい写像
`y = r·x − (r−1)·N/2` (§3.5) に修正し、オンセットが基準と完全一致した。

### 5.2 全 bin 位相リセットによる持続音の劣化

Goertzel フィルタによるコヒーレンス計測 (持続音の長窓振幅) で、和音の振幅が基準の
1/5〜1/20 に落ちることを発見。エネルギー自体 (RMS) は保たれており、正体は
「ドラムのたびに全 bin リセット → 持続音の位相が 0.25〜0.5 秒ごとにランダムジャンプ」。
リセットを立ち上がり領域に限定する方式 (§3.4) に変更し、基準と同等のコヒーレンスになった。

### 5.3 FFT 4096 で近接音が −6dB 落ちる

修正後も和音の 1 音 (E4) だけが恒常的に −6dB 低いことを発見。原因は短 3 度間隔
(C#4 277Hz と E4 330Hz、4096 点では 4.5 bin 差) で **Hann メインローブ (4 bin 幅) が
領域境界をまたいで分割**され、ローブの半分が隣のピークの回転を受けて時間平均で
打ち消されること。N=8192 (bin 間隔 5.9Hz) に変更して解消。トランジェントへの影響は
アタック傾斜・キック毎エネルギーの計測で同等を確認した (領域限定リセットが効いている)。

## 6. 検証方法

1. **ユニットテスト** (`Tests/PitchEngineTests.cpp`): 周波数 (ゼロ交差法 ±3Hz)・振幅・
   尺の厳密一致・ステレオ独立・0 半音のビット同一・無音・キャンセル・進捗単調性・
   WAV 往復・FFT ラウンドトリップ
2. **コヒーレンス計測** (品質変更時に必須): 合成曲に対し Goertzel フィルタで
   シフト後の各持続音の長窓振幅を測り、劣化がないこと (位相ジャンプはコヒーレント
   積分の低下として現れる。RMS では検出できない)
3. **聴感 A/B**: `UtawavePitchTool <audiofile> <semitones>` で実カラオケ音源を変換し、
   ドラムのアタック / ボーカルの質感 / リバーブの尾 / ステレオ定位を聴き比べる

## 7. 既知の制限と改善候補

- **フォルマント保持なし**: 全帯域を等しくシフトする (DAW のキー変更用途では
  オケ全体を動かすため意図どおり)。ボーカル単体の自然なキー変更には
  フォルマント補正 (スペクトル包絡の分離) の追加が必要
- **単一解像度 STFT**: 8192 固定。半音間隔以下の密集した低域クラスタや極端に速い
  パッセージではマルチ解像度 (帯域別の窓長) に分がある
- **メモリ**: 全体をメモリ展開して処理する (5 分のステレオ 48kHz で数百 MB)。
  ストリーミング化は構造上可能だが未実装
- 推奨レンジ ±6 半音。±12 でも破綻はしないがアーティファクトが増える
